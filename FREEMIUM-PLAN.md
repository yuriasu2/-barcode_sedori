# フリーミアム化 実装企画書（無料版 / Pro版）

作成: 2026-07-14 / 最終更新: 2026-07-14（未決事項を全て決定） / ステータス: **仕様確定・実装待ち**

## 1. 目的

- 無料版で体験を提供しつつ、広告収入とPro課金で収益化する。
- SP-API / Keepa の従量コスト（トークン・レート制限）を無料ユーザーから守る。

## 2. プラン定義（機能マトリクス）

**重要な前提**: SP-APIは**利用者自身のセラーアカウントをOAuthで接続（BYO）**して使う。レート枠はセラー単位のため利用者間で干渉せず、SP-APIのオファーは第1段階（/api/search）に同梱済み＝**追加通信ゼロ**。よって無料でもSP-API接続者にはオファーを開放する。守るべき共有コストは**サーバーのKeepa鍵**のみ。

| 機能 | 無料版 | Pro版（課金） |
|---|---|---|
| バーコードスキャン | ○（**5秒に1回**まで） | ○（現行1秒クールダウン） |
| 1日のスキャン上限 | **100件/日** | 無制限 |
| OCRスキャン（ISBN/JAN文字認識） | ×（トグルはロック表示→タップでペイウォール） | ○ |
| SP-API接続（自分のセラーをOAuth接続=BYO） | ○ | ○ |
| 第1段階: 簡易価格（タイトル/画像/新品・中古最安） | ○（SP-API接続時はSP-API、未接続はKeepa） | ○ |
| 第2段階: オファー一覧（コンディション・送料込・最安順） | **SP-API接続時: ○**（同梱＝追加コストなし） / **Keepaのみ（未接続）: ×**（鍵＋ぼかしダミー） | ○ |
| Keepaグラフ | ×（**ぼかしグラフ＋鍵＋広告バナー**） | ○ |
| 広告 | あり（グラフ枠に表示） | なし |
| スキャン履歴保存 | 無制限（制限なし） | 無制限 |
| 損益分岐点（breakEven表示） | ○（SP-API接続時。各自の枠で算出＝共有コストなし） | ○ |

無料層は実質2種類に分かれる:
- **BYO-SP-API接続者**（実セラー）: 価格＋オファー一覧＋breakEven まで見られる（各自の枠）。Pro差分は OCR / Keepaグラフ / 広告なし / スキャン速度・日次無制限。
- **Keepaのみ（未接続）**: 第1段階の簡易価格まで。オファーは鍵＋ぼかしダミー、グラフはロック。

補足（コンバージョン設計）:
- グラフ枠は「広告だけ」より「ロックされた価値が見える」方が課金転換率が高いため、**ダミー/ぼかしグラフ＋鍵アイコン＋その下に広告** の構成とする。
- **Keepa経路のオファー**（＝SP-API未接続の無料ユーザー）は**実データを一切取得しない**（Keepa第2段階=offers付きgetProductはトークン消費が大きいため）。代わりに**ハードコードのダミーオファー数行にぼかし（blur）＋鍵アイコン**を重ねて表示し、タップでペイウォールへ。ダミーなので**サーバーのKeepaトークン消費はゼロ**。
- 一方 **SP-API経路のオファーは第1段階に同梱済み**なので、無料でもそのまま実データを表示する（追加コストなし）。

## 3. 課金方式

- **月額サブスクリプション（StoreKit 2 / 自動更新）**。買い切りは提供しない。
  - 理由: SP-API/Keepaの利用コストが毎月発生するため、買い切りだとヘビーユーザーが恒常的な赤字要因になる。
- **価格: ¥1,980/月**。**無料トライアル: 3日間**（導入オファー）。
- 必須実装: 購入 / 復元（Restore Purchases）/ 期限切れ検知 / ペイウォール画面。

## 4. アーキテクチャ方針

### 4.1 エンタイトルメント（プラン状態）の持ち方

- iOS: `EntitlementStore`（新規, ObservableObject）を追加。StoreKit 2 の `Transaction.currentEntitlements` を起動時+変更時に監視し、`isPro: Bool` を公開。
- 全ゲートはこの `isPro` を単一の真実として参照する（散在させない）。

### 4.2 サーバー側ゲート（重要）

クライアント側の出し分けだけでは改ざん・直叩きで回避可能。段階的にサーバーでも制限する。ゲートは**ソース依存**（SP-APIは各自の枠なので開放、Keepaは共有コストなので制限）。

- アプリはリクエストに `X-App-Plan: free|pro` を付与（Phase 1: 自己申告）。
- サーバーの無料プラン動作:
  - `/api/search`: **SP-API接続時（refresh tokenあり）はSP-API＝オファー同梱を含めて通常どおり返す**（各自の枠のため制限不要）。**SP-API未接続（Keepaフォールバック）時は第1段階の簡易価格のみ**返し、オファーは含めない。
  - `/api/offers`: **Keepa経路（`source=keepa`）は無料に 403（`plan_required`）**。SP-APIのオファーは第1段階同梱のためこのエンドポイント自体を無料が呼ぶ必要がない（アプリは既にsource=spapi時に呼ばない実装）。
  - `/api/graph`: 無料は 403（`plan_required`）。※グラフはKeepa鍵消費＝共有コストのためPro限定。
  - Keepa経路のキャッシュTTLを長めに（未接続の無料はやや古い価格で許容し、サーバーのKeepaコストを削減）。
- Phase 2 で自己申告を廃し、**App Store Server API によるレシート/トランザクション検証**をサーバーで行う（署名付きトークンをアプリへ返し、以降のAPIはそのトークンで判定）。

### 4.2b SP-API認証情報（BYO / OAuth）の保存方針

- 利用者は自分のセラーアカウントを **OAuth（LWA）で接続**し、取得した refresh token をアプリが保持、リクエストヘッダー `X-Spapi-Refresh-Token` で送る（clientId/clientSecret は開発者アプリ共通=サーバー .env）。→ 現行実装済み。
- **公開前に、試験用の「手動SP-APIキー入力欄」は削除**し、接続導線はOAuthのみにする。

#### 保存方式は2段階で考える

**Stage A: 端末セキュア保存（当面の推奨・低コスト）**
- refresh token は現状 `UserDefaults` 保存（`SettingsStore.swift`）だが、セラーアカウントのアクセス権を持つ機微情報のため、**端末のセキュアストレージへ移行する**。
  - iOS: **Keychain**（`kSecClassGenericPassword` / `kSecAttrAccessibleWhenUnlockedThisDeviceOnly` / 非同期＝Synchronizable=false）。
  - Android（将来）: **Android Keystore で生成した鍵で暗号化 → EncryptedSharedPreferences もしくは DataStore+Tink に保存**（`security-crypto` はメンテ停滞のため Tink 直利用を優先検討）。`setUnlockedDeviceRequired(true)`、端末バインド。
- 受け渡し経路も可能なら **ASWebAuthenticationSession（iOS）/ Custom Tabs（Android）** を使い、カスタムURLスキームのクエリでトークンを渡さない設計にする。
- iOS単独 / 少数ユーザーの段階はこれで十分。

**Stage B: サーバー保管＋セッショントークン（マルチプラットフォーム本格化時に移行）**
- refresh token を**端末に置かずサーバーに暗号化保存**し、アプリには**失効可能・権限限定のセッショントークン**のみ渡す（端末には引き続きセッショントークンは保存するが、マスター資格情報ではない＝漏洩時の被害限定＋サーバー側で即失効可）。
- 必要になるもの:
  - **識別＋セッション管理**（必ずしもID/PWログインではない。最小は匿名デバイスID＋アプリ発行セッション。**機種変更/再インストール復元や iOS↔Android の連携共有**が要るなら **Sign in with Apple / Google Sign-In** を導入）。
  - サーバーに**ユーザー識別＋永続ストレージ（DB／暗号化保存）**（現状の依存ゼロ・インメモリ構成には無いので新設）。
  - **セッション期限切れ処理**（401→黙ってセッション再発行→元リクエストを自動リトライ→再発行も失敗なら再認証誘導）。
  - **OAuth完了の冪等化**（連携中に通信断でも「トークン保存済みだがセッション未受領」等の中途半端が残らないよう再開可能に）。
- 通信面の注意: 本アプリは元々検索が全面サーバー依存（オフライン不可）で、Stage B でも**追加ホップは無し**（同一サーバーが認証＋API担当）。新たな失敗モードはセッション期限切れのみで、上記リトライ設計で吸収できる。

#### 移行トリガー（このいずれかで Stage A → B へ）
- Android版を本格展開し、Pro/連携状態を iOS↔Android で共有したくなった。
- 機種変更・再インストール後に Amazon 再連携なしで復元する体験を提供したくなった。
- セラー資格情報の端末保存リスクを完全に無くしたくなった。

### 4.3 スキャン制限

- `ScannerView` の `emitCooldown`（現行 1.0 秒固定）を外から注入可能にし、無料=5.0秒 / Pro=1.0秒。
- クールダウン中に読み取った場合は「あと◯秒」の小さなオーバーレイ表示（無反応だと故障に見えるため）。
- **日次上限: 無料は100件/日**。クライアントでカウント（日付が変わればリセット、タイムゾーンはデバイスローカル）し、超過時はペイウォール誘導。サーバーでもデバイスID単位のレート制限をバックストップとして入れる（Phase 3）。

### 4.4 広告

- グラフ枠に AdMob バナー（アダプティブバナー）を表示。Pro は非表示＋実グラフ。
- 必要対応: AdMob SDK 導入（XcodeGen `project.yml` に依存追加）、ATT（トラッキング許可）ダイアログ、`PrivacyInfo.xcprivacy`（プライバシーマニフェスト）、App Store 審査のデータ収集申告。
- 注意: 本アプリ初の外部SDK依存。広告なし構成（ロック表示のみ）でまずリリースし、広告を後続フェーズに回す選択肢もある。

## 6. 実装フェーズ

- **Phase 1（クライアント課金 + 自己申告ゲート）**
  1. StoreKit 2 / EntitlementStore / ペイウォール（¥1,980/月・3日トライアル・復元）
  2. クライアント側ゲート（OCRロック・5秒クールダウン・日次100件・オファーぼかしダミー＋鍵・グラフロック表示）
  3. サーバー側ゲート（`X-App-Plan` ベース・ソース依存: SP-APIは開放／Keepaの offers・graph は 403）
  4. テスト（サーバー）+ 実機確認 → リリース
- **Phase 2（広告）**
  5. AdMob 導入（ATT・プライバシーマニフェスト・審査申告）、グラフ枠へバナー表示
- **Phase 3（不正対策強化）**
  6. App Store Server API によるサーバー側レシート検証（自己申告ヘッダー廃止）
  7. サーバー側のデバイス単位レート制限（日次上限のバックストップ）

## 7. リスク・注意

- **審査**: サブスクは「機能比較・価格・解約方法」の明記が必要。トライアル文言も規約に沿うこと。
- **広告SDK**: ATT許可率次第で収益が変動。導入コスト（SDK・審査対応）に対し、少DAU期はPro課金の方が費用対効果が高い可能性 → 広告をPhase 2に分離した理由。
- **無料のKeepa消費**: SP-API未接続の無料ユーザー増加はKeepaトークン消費に直結。日次上限とKeepaキャッシュTTL延長を同時に入れるのが安全（SP-API接続者は各自の枠なので対象外）。
- **SP-APIレート枠**: BYO（セラー単位）のため利用者間で干渉しない。共有コストにならない。
- **既存ユーザー**: 現行TestFlight/自分用ビルドはすべてPro相当の動作。公開時の初期プラン扱いを決めること。
