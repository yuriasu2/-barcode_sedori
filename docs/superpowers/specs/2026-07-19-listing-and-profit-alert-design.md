# アマレンズ 出品機能・利益アラート 設計書

- 作成: 2026-07-19 / 策定: Fable5（ブレインストーミング承認済み）
- 状態: **設計承認済み・実装未着手**（ユーザー指示により実装は保留）
- 対象リポジトリ: バーコードせどり (1)（iOSアプリ + server）

## 背景と決定事項

Amazon販売パートナーアプリストアへの掲載申請準備中に、2つの新機能を追加する方針が決まった。

| 項目 | 決定 |
|---|---|
| 利益アラート | スキャン時の即時判定。**仕入れ値は使わず損益分岐点で判定** |
| 出品 | 仕入れリストに貯めて後で出品。**自己発送（MFN）のみ**。アプリ内で完結 |
| 課金区分 | **両方ともPro限定** |
| SKU | 自動生成＋編集可 |
| 出品価格の初期値 | 同コンディション最安値（送料込み） |
| コンディション説明文 | テンプレート＋編集可 |
| 出品制限チェック | 出品画面表示時に事前チェック |
| SP-APIロール | Product Listing ロールは**承認済み**（PII非該当の通常ロール） |

## フェーズ分け（案A: 承認済み）

- **Phase 1: 利益アラート＋仕入れリスト** — SP-API追加なし。先行リリース可能
- **Phase 2: アプリ内出品** — Listings系API・サーバー拡張あり

理由: 利益アラートは既存コードの小さな拡張で済むのに対し、出品は新API群・
エラー処理・不可逆操作の確認UIを要する別次元の作業。混ぜると軽い機能が重い機能に引きずられる。

## Phase 1a: 利益アラート（Pro限定）

- 設定に「利益ライン」（円）を追加。`SettingsStore` に保存
- スキャン結果表示時、**損益分岐点 ≥ 利益ライン**なら結果カードを強調表示
  （緑バナー＋ハプティクス振動）
- 既存の損益分岐点計算をそのまま使うため**API追加なし**
- 無料ユーザーが設定画面で触ると Paywall へ誘導

## Phase 1b: 仕入れリスト

- 商品詳細に「仕入れリストへ追加」ボタン（Pro限定）
- プレースホルダの仕入れタブ（`PurchaseTabView`）を実体化:
  追加した商品の一覧・削除・出品への導線
- 保存は**端末ローカルのみ**（スキャン履歴と同じ方式）。サーバーには置かない
  （プライバシーポリシー「履歴は端末内のみ」と整合させるため）

## Phase 2: 出品（Pro＋SP-API連携必須）

### 出品フォーム（仕入れタブの商品 →「出品」）

| 項目 | 仕様 |
|---|---|
| コンディション | 選択式（ほぼ新品／非常に良い／良い／可） |
| 価格 | 初期値＝同コンディション最安値（送料込み）。画面表示時に再取得 |
| SKU | 自動生成 `AMLZ-日付-連番`、編集可 |
| 説明文 | 設定画面で編集できるコンディション別テンプレートを自動適用、個別編集可 |
| 数量 | 既定1 |

### フロー

1. 出品画面表示時に **Listings Restrictions API** で出品制限をチェック。
   制限ありなら入力させず、Amazonの解除申請リンクを案内
   （解除申請自体はAPI経由で不可。セラー自身がSeller Centralで手続き）
2. 出品実行前に**確認ダイアログ必須**（不可逆操作のため）
3. 実行は既存のBYO機構を流用: サーバーに `X-Spapi-Refresh-Token` を送り、
   サーバーが `putListingsItem` を呼ぶ
   - 既存ASINへのオファー出品。`PRODUCT` product type を使用（新規ASIN作成はしない）
   - **トークンも出品内容もサーバーに保存しない**（DPP整合）
4. 出品は非同期受理のため「出品を受け付けました。反映まで数分かかります」と表示し、
   リスト項目に「出品済み」マークを付ける

### サーバー追加API

- `GET /api/listings/restrictions?asin=` — 出品制限の確認
- `POST /api/listings` — オファー出品（putListingsItem）
- 両方とも **Proヘッダー＋リフレッシュトークン必須**で403ゲート
  （既存 `isProRequest` を流用）

### エラー処理

- 出品失敗（価格不正・制限・トークン失効）はサーバーのエラー本文をそのまま表示し、
  フォームに戻ってリトライ可能
- 同一SKUへの再実行は `putListingsItem` の仕様上、上書きになるため二重出品にはならない

## Amazon申請への影響（調査済み・出典は当時の公式ドキュメント）

- Product Listing ロールは Restricted ではなく PII も伴わない
  → DPPのPII向け追加要件は原則発生しない
- Developer Console のアプリにロールを追加すると、
  **連携済みユーザー全員が再認可（Re-Authorize）必要**になる
  （"anytime you add a role to your application"）。
  現在は開発者本人の検証用連携のみなので、**Phase 2 着手時にロールを追加して再連携**すれば済む
- 提出予定の掲載内容（APPSTORE-LISTING-DRAFT.md）は**変更しない**。
  実装済み機能のみ記載する方針を維持し、出品機能の追記は実装完了後に
  「Edit Listing」で行う（公開後も編集可能・編集中も掲載は公開継続、と確認済み）
- 関連API: Listings Items API / Listings Restrictions API / Product Type Definitions API
  （すべて Product Listing ロール配下）

## テスト方針

- Phase 1: シミュレータで判定閾値の境界（未満/一致/超過）と無料/Proのゲートを確認
- Phase 2: SP-APIサンドボックスまたは開発者アカウントの実出品→即時削除で
  putListingsItem の受理と反映を確認。出品制限ありASINで案内表示を確認
- 既存の標準ルール（変更ごとにシミュレータで動作確認）に従う

## 実装しない・やらないこと（YAGNI）

- FBA納品プラン（自己発送のみ）
- 新規ASIN（カタログ新規作成）
- 登録商品の相場監視・プッシュ通知（利益判定はスキャン時のみ）
- 出品内容のサーバー保存・履歴のクラウド同期
